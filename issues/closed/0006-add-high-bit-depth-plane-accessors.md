# DecodedFrame にハイビット深度用のプレーンアクセサを追加する

Model: Opus 4.6

## 概要

現在の `y_plane()`, `u_plane()`, `v_plane()` は常に `&[u8]` を返す。
10-bit や 12-bit コンテンツの場合、1 ピクセルあたり 2 バイト (リトルエンディアン) で格納されているが、
ユーザーが自分でバイト列を `u16` に変換する必要がある。
これは不便であり、エンディアン変換の誤りなどバグの温床になる。

## 追加すべきメソッド

```rust
impl DecodedFrame {
    /// フレームの Y 成分を u16 スライスとして返す
    ///
    /// ビット深度が 8 の場合は None を返す
    pub fn y_plane_u16(&self) -> Option<&[u16]> { ... }

    /// フレームの U 成分を u16 スライスとして返す
    ///
    /// ビット深度が 8 の場合、またはピクセルレイアウトが I400 の場合は None を返す
    pub fn u_plane_u16(&self) -> Option<&[u16]> { ... }

    /// フレームの V 成分を u16 スライスとして返す
    ///
    /// ビット深度が 8 の場合、またはピクセルレイアウトが I400 の場合は None を返す
    pub fn v_plane_u16(&self) -> Option<&[u16]> { ... }
}
```

## 実装方針

- `is_high_depth()` が false の場合は `None` を返す
- `std::slice::from_raw_parts` で `*const u8` を `*const u16` にキャストしてスライスを構築する
- dav1d はネイティブエンディアンで格納するため、エンディアン変換は不要
- スライスの長さはバイト数 / 2 になる

## 根拠

dav1d は 10-bit / 12-bit AV1 コンテンツのデコードに対応しており、
HDR コンテンツ (BT.2020 + PQ) は一般的に 10-bit である。
ハイビット深度のピクセルデータを `&[u16]` として直接アクセスできないと、
全てのユーザーが同じ unsafe な変換コードを書くことになる。
安全な API としてライブラリ側で提供すべきである。

## 解決方法

`DecodedFrame` に `y_plane_u16()`, `u_plane_u16()`, `v_plane_u16()` メソッドを追加した。
ビット深度が 8 の場合は `None` を返し、I400 の場合は U/V で `None` を返す。
